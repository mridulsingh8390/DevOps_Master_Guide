# K9s Complete Guide (Kubernetes Terminal UI)
## Fast Kubernetes Operations for DevOps Engineers

> This file is practical and operator-focused.
> Covers install, navigation, useful workflows, plugins, troubleshooting.

---

## 1) What is K9s and Why it Matters

K9s is a terminal-based UI for Kubernetes that helps you inspect and manage cluster resources faster than raw kubectl for many day-to-day tasks.

## Why use it?
- Faster navigation across namespaces/resources
- Quick logs, describe, exec, delete, edit
- Better operational visibility during incidents
- Lower cognitive load vs many repeated kubectl commands

---

## 2) Install K9s (Ubuntu)

Option A (snap):
```bash
sudo snap install k9s
```

Option B (binary):
```bash
cd /tmp
wget https://github.com/derailed/k9s/releases/latest/download/k9s_linux_amd64.deb
sudo apt install -y ./k9s_linux_amd64.deb
```

Verify:
```bash
k9s version
```

---

## 3) Pre-Checks Before Running

- Ensure `kubectl` works:
```bash
kubectl get nodes
kubectl config current-context
```

Then launch:
```bash
k9s
```

K9s uses your current kubeconfig context.

---

## 4) Core Navigation Basics

Inside K9s:

- `:` command mode (like Vim-style command entry)
- `?` help
- `/` filter current list
- `Esc` back
- `Ctrl+c` quit

Common resource jumps:
- `:po` Pods
- `:deploy` Deployments
- `:svc` Services
- `:ing` Ingress
- `:ns` Namespaces
- `:node` Nodes
- `:cm` ConfigMaps
- `:sec` Secrets
- `:sts` StatefulSets
- `:ds` DaemonSets
- `:job` Jobs
- `:cronjob` CronJobs

---

## 5) Namespace and Context Switching

### Switch namespace
- `:ns`
- select namespace and press Enter

### Switch context
- `:ctx`
- choose kube context

This is very useful for multi-cluster operations.

---

## 6) Resource Actions (Most Used)

Select resource row, then:

- `d` describe
- `l` logs
- `s` shell/exec (if available)
- `e` edit resource YAML
- `y` YAML view
- `x` delete resource
- `Shift+f` port-forward (for supported resources)

Logs follow mode:
- `l` then `f` (depends on mapping/version)
- Use filters while viewing logs

---

## 7) Incident Workflow with K9s (Recommended)

1. `:po -n <ns>` and sort by restarts/status  
2. Open failing pod logs (`l`)  
3. Describe pod (`d`) for events and probe failures  
4. Jump to deployment (`:deploy`) and inspect rollout  
5. Check service/endpoints (`:svc`, `:ep`)  
6. If needed, exec shell (`s`) for live debugging  

This is much faster than many separate kubectl calls.

---

## 8) Useful Views You Should Use Daily

- `:xray po` → workload relationship exploration
- `:events` → cluster events timeline
- `:top` → live CPU/memory (if metrics-server installed)
- `:pulses` → cluster health summary
- `:helm` → Helm releases (if enabled/support present)

## 9) K9s Configuration and Customization

Config path (Linux):
- `~/.config/k9s/config.yaml`

Skin/themes path:
- `~/.config/k9s/skins/`

You can customize:
- refresh rates
- UI skin/theme
- aliases
- logger buffer sizes

---

## 10) Plugins (Power Feature)

K9s supports custom plugins to execute scripts/commands on selected resources.

Plugin config path:
- `~/.config/k9s/plugins.yaml`

Example idea:
- Open `kubectl describe` in pager
- Trigger custom diagnostics script
- Run security checks for selected namespace

---

## 11) Security and Safety Tips

1. Be careful with delete (`x`) in production clusters  
2. Use RBAC to restrict what users can do  
3. Keep separate kubecontexts for prod vs non-prod  
4. Set clear context names to avoid mistakes  
5. Prefer read-only access for broad team visibility roles  

---

## 12) Troubleshooting

## K9s opens but no resources
- wrong context/namespace
```bash
kubectl config current-context
kubectl get ns
```

## Permission errors
- RBAC denies actions
```bash
kubectl auth can-i list pods -n <ns>
```

## Top view empty
- metrics-server missing or unhealthy
```bash
kubectl get pods -n kube-system | grep metrics
```

## Slow UI
- very large cluster + short refresh
- tune refresh settings in config

## 14) kubectl vs K9s Quick Mapping

- `kubectl get pods -A` -> `:po -A`
- `kubectl describe pod ...` -> select pod + `d`
- `kubectl logs -f ...` -> select pod + `l`
- `kubectl edit deploy ...` -> select deploy + `e`
- `kubectl exec -it ... -- sh` -> select pod + `s`
